# [기능 명세] 인증 페이지 및 공통 레이아웃 (Auth & Common Layout)

본 문서는 사용자의 비밀번호 없는 로그인 화면과 모바일 대응을 위한 전역 웹 셸(Web Shell) 레이아웃의 상세 사양을 정의합니다.

---

## 1. 비밀번호 없는 OTP 로그인 화면 (Auth Page)

### 1.1 UI/UX 개요
- **대상**: 관리자, 현장 스태프, 전문가, 스타트업 공통 로그인 관문.
- **특징**:
  - 전문가와 스타트업은 등록된 이메일 또는 휴대전화로 전달되는 6자리 **일회용 인증번호(OTP)**로 접근합니다.
  - 운영본부는 참가 기업별 비밀 코드를 안내하지 않고, 모든 참가자에게 동일한 로그인 URL과 사용 절차를 한 번에 안내합니다.
  - 관리자와 현장 스태프는 Supabase Auth 이메일·비밀번호를 사용합니다.

### 1.2 화면 구성 요소 (Wireframe Spec)
- **로고 영역**: 최상단에 서비스 로고 및 행사 타이틀 표시.
- **역할 선택 탭 (Optional)**:
  - 탭 구성: `참가 스타트업` | `전문가/자문위원` | `운영진` (선택된 역할에 따라 입력 필드와 가이드 메시지 변경).
- **입력 필드**:
  - 참가자 1단계: `등록 이메일 또는 휴대전화 번호`.
  - 참가자 2단계: `6자리 인증번호`.
  - 운영진: `이메일 주소`, `비밀번호`.
- **로그인 버튼**:
  - 활성화 조건: 모든 필드 값이 채워졌을 때 활성화.
  - 클릭 시 로딩 애니메이션(스피너) 표시.

### 1.3 로그인 로직 및 에러 처리
- **인증 처리**:
  1. 참가자가 등록 이메일 또는 휴대전화 번호를 제출합니다.
  2. 서버는 계정 존재 여부를 노출하지 않는 일반 응답을 반환하고, 일치하는 활성 참가자가 있으면 선택 채널로 6자리 OTP를 비동기 발송합니다.
     - 입력값이 여러 활성 참가자와 일치하면 임의 계정으로 발송하지 않고 일반 응답만 반환합니다.
  3. 참가자가 OTP를 제출하면 서버가 해시, 만료 시각, 시도 횟수, 사용 여부를 검증합니다.
  4. 검증 성공 시 OTP를 즉시 사용 처리하고 참가자 커스텀 JWT를 발급합니다.
  5. 운영진은 Supabase Auth로 인증합니다.
  6. 인증 성공 시 사용자의 `role`에 따라 해당 대시보드로 리다이렉트합니다.
     - `ADMIN` $\rightarrow$ `/admin/events`
     - `STAFF` $\rightarrow$ `/staff/check-in`
     - `EXPERT` $\rightarrow$ `/expert/dashboard`
     - `STARTUP` $\rightarrow$ `/startup/booking`
- **에러 핸들링**:
  - OTP 요청 완료 시 계정 존재 여부와 무관하게 `"등록된 연락처가 확인되면 인증번호를 발송했습니다."`를 표시합니다.
  - OTP 불일치·만료·사용 완료 시 `"인증번호가 올바르지 않거나 만료되었습니다. 새 인증번호를 요청해 주세요."`를 표시합니다.
  - 네트워크 오류 시: `"네트워크 연결 상태를 확인하고 다시 시도해 주세요."` 문구 노출.

### 1.4 OTP 발급 및 운영 규칙
- OTP는 숫자 6자리이며 기본 유효시간은 5분입니다.
- 원문은 발송에만 사용하고 DB와 애플리케이션 로그에는 남기지 않습니다. DB에는 해시, 대상 사용자, 채널, 만료 시각, 시도 횟수, 사용 시각을 저장합니다.
- 같은 대상의 재요청은 60초 이후 허용하고, 챌린지당 검증 실패는 최대 5회로 제한합니다.
- 새 OTP를 발급하면 같은 목적의 이전 미사용 OTP는 무효화합니다.
- 발송은 이메일을 기본 채널로 하고, 휴대전화 번호가 검증·등록된 경우 알림톡/SMS를 대체 채널로 지원합니다.
- 로그인 세션은 기본 12시간으로 하되 행사 종료 시각을 넘지 않도록 설정합니다. 관리자는 `session_version` 증가를 통해 기존 참가자 세션을 강제 무효화할 수 있습니다.
- 공통 행사코드를 제공할 수 있으나 행사 선택용 보조값일 뿐, OTP를 대신하는 인증 수단으로 사용하지 않습니다.
- 현장 발송 장애 시 관리자는 본인 확인 후 짧은 만료시간의 1회용 로그인 링크를 발급할 수 있으며 처리 이력을 남깁니다.

## 2. 반응형 공통 레이아웃 (Common Shell Layout)

### 2.1 UI/UX 가이드라인 (Tailwind CSS 전용)
- **모바일 퍼스트**: 현장에서 스마트폰이나 태블릿으로 주로 조작하므로 모바일 뷰포트(360px ~ 768px)를 기본 설계 기준으로 삼습니다.
- **인라인 스타일 금지**: 스타일링은 100% Tailwind 클래스를 이용합니다.
- **지정 컬러 시스템**:
  - 브랜드 강조색 (Primary Accent): `#E22213` (선명한 레드) $\rightarrow$ 핵심 조작 버튼, 하이라이트 아이템에 전면 적용.
  - 베이스 무채색 (Neutral Base): `#515151` (다크 그레이) $\rightarrow$ 헤더, 사이드바 배경, 텍스트 색 및 카드 테두리에 배치.
- **고대비 가시성 (흐리멍텅한 UI 배제)**:
  - 카드·입력창·탭에 동일한 굵은 테두리를 반복하지 않습니다.
  - 일반 카드: 흰색 배경, 12~16px 모서리, 1px 중립 회색 경계선, 필요한 경우에만 매우 약한 그림자.
  - 입력창: 기본 1px 중립 경계선, 포커스 시 브랜드 레드 링 또는 경계선.
  - 역할 탭: 굵은 칸막이 대신 연회색 컨테이너 안에서 선택된 항목의 배경과 글자색을 전환하는 세그먼트 컨트롤.
  - Primary 버튼: 브랜드 레드 면을 사용하고 별도 진회색 테두리를 두지 않습니다.
  - 오류/안내: 굵은 사각 테두리 대신 연한 상태 배경, 아이콘 또는 왼쪽 강조선을 사용합니다.

### 2.2 좌측 메뉴 네비게이션 구조 (Left Navigation Layout)
- **데스크톱**: 화면 좌측에 240px 고정 영역으로 사이드바 메뉴가 상시 표출됩니다.
- **모바일/태블릿**: 화면 좌측 상단의 햄버거 메뉴를 탭하면, 사이드바가 좌측에서 우측으로 부드럽게 슬라이드인(Slide-in) 형태로 들어오는 토글 메뉴를 지원합니다.


### 2.3 상단 헤더 (Top Header)
- **디바이스 대응**: 데스크톱에서는 고정 헤더, 모바일에서는 좌측 햄버거 메뉴 포함.
- **표시 정보**:
  - 왼쪽: 햄버거 메뉴 버튼(모바일 전용), 서비스 로고.
  - 오른쪽: 현재 로그인한 사용자의 프로필 아이콘, 유저명(`예: 홍길동 대표님`), 로그아웃 버튼.
- **실시간 남은 시간 표시 (행사 진행 중인 경우)**:
  - 현재 활성화된 세션의 카운트다운 타이머가 헤더 중앙에 눈에 띄는 배지 형태로 상시 표시됨.

### 2.4 토글형 네비게이션 사이드바 (Sidebar)
- **동작**:
  - 데스크톱: 좌측 240px 너비로 고정 노출.
  - 모바일/태블릿: 햄버거 버튼 클릭 시 좌측에서 슬라이드인(Slide-in) 형태로 등장하며 뒷배경에 반투명 오버레이(Backdrop) 추가. 오버레이 또는 닫기 버튼 클릭 시 슬라이드아웃.
- **메뉴 목록 (유저 역할별 가변 제공)**:
  - **관리자**: 행사 목록(`admin/events`), 참가자 DB 관리(`admin/users`), 설정.
  - **현장 스태프**: 출석 체크(`staff/check-in`).
  - **전문가**: 오늘의 스케줄(`expert/dashboard`), 이전 상담 이력.
  - **스타트업**: 내 예약 관리(`startup/booking`), 안내 사항.

## 3. 코드 설계 가이드라인 (개발 준수 사항)
- **최대 코드 라인**: `auth.jsx`, `Layout.jsx` 등의 핵심 파일은 컴포넌트를 서브 유닛(`LoginForm.jsx`, `Header.jsx`, `Sidebar.jsx`)으로 세분화하여 각 파일당 **500줄 제한**을 철저히 준수합니다.
- **인증 세션 상태 관리**: React Context API (`AuthContext.js`) 또는 Zustand를 이용하여 전역 세션 상태를 저장하며, 페이지 전환 시마다 Route Guard를 거치게 설정합니다.
- **보안 준수**: 상세 RLS와 RPC 정책은 [인증·권한·트랜잭션 정책](./security_transactions.md)을 따릅니다.
