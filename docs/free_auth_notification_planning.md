# 무료 운영 전환 기획 메모: 참가자 로그인과 알림

> 작성일: 2026-06-26  
> 상태: 미확정 기획 메모. 아직 구현 기준 문서가 아니며, Phase 7 또는 인증 정책 변경 전 의사결정 참고용으로 보관한다.

## 1. 배경

현재 참가자 인증은 `등록 이메일/휴대전화 OTP 요청 -> 6자리 OTP 검증 -> 커스텀 JWT 발급` 구조다.

이 구조는 보안과 사용자 경험 측면에서는 좋지만, 실제 운영에서 이메일/SMS/알림톡 자동 발송을 붙이면 외부 발송 서비스 비용과 운영 계정 설정이 필요하다. 전면 무료 운영을 우선할 경우, 로그인 자체가 발송 API에 의존하지 않도록 기획을 바꾸는 방안을 검토한다.

## 2. 현재 확인한 비용 구조

### Supabase

- Supabase는 이메일 OTP, 휴대폰 OTP 로그인을 지원한다.
- 단, 운영용 이메일은 Custom SMTP 설정이 사실상 필요하다. Supabase 기본 SMTP는 운영용이 아니며 강한 제한이 있다.
- 휴대폰 OTP는 SMS provider 설정이 필요하다. 공식 문서 기준 MessageBird, Twilio, Vonage, Textlocal 계열을 지원하며, 국내 카카오 알림톡/Solapi는 내장 provider로 보기 어렵다.
- Supabase 과금은 별도다. Auth를 직접 쓰면 MAU, Edge Function을 쓰면 호출량 등이 Supabase 사용량으로 계산된다.

참고:
- Supabase Phone Login: https://supabase.com/docs/guides/auth/phone-login
- Supabase Email OTP: https://supabase.com/docs/guides/auth/auth-email-passwordless
- Supabase Custom SMTP: https://supabase.com/docs/guides/auth/auth-smtp
- Supabase Billing: https://supabase.com/docs/guides/platform/billing-on-supabase

### 외부 발송 서비스

Solapi 공개 가격표 기준:

| 채널 | 단가 |
|---|---:|
| 카카오 알림톡 | 13원/건, VAT 별도 |
| SMS | 18원/건, VAT 별도 |
| LMS | 45원/건, VAT 별도 |
| MMS | 110원/건, VAT 별도 |

비용만 보면 카카오 알림톡이 SMS보다 저렴하다. 다만 카카오 비즈니스 채널, 알림톡 템플릿 등록/승인, 실패 시 SMS fallback 같은 운영 조건이 붙는다.

참고:
- Solapi 가격표: https://solapi.com/pricing

## 3. 전면 무료 운영 시 포기하거나 보류할 기능

전면 무료 운영은 핵심 매칭/예약/상담 기능을 포기한다는 뜻이 아니라, 자동 발송 기반 기능을 보류한다는 뜻에 가깝다.

보류 대상:

- SMS OTP 로그인
- 카카오 알림톡 OTP 로그인
- 이메일 OTP 자동 발송, 운영 SMTP까지 완전 무료로 보장하려면 이메일 OTP도 보류
- 예약 리마인드 자동 발송
- 미예약 스타트업 알림 재발송
- 행사 시작/변경/취소 자동 안내 발송
- 발송 실패 추적, 재시도, 백오프, 발송 로그 대시보드
- 알림톡 템플릿 관리/승인 연동
- 문자 발송 비용/쿼터 관리

그대로 유지 가능:

- 관리자/스태프 Supabase Auth 로그인
- 참가자/전문가 포털
- 예약/변경/취소
- AI 배치, 강제 배정/취소
- 출석 처리
- 상담 시작/상담일지/공개 코멘트
- 만족도 설문
- CSV/엑셀성 내보내기
- 현장용 1회용 로그인 링크
- 화면 기반 QR/현장 체크인

## 4. 대안: 이름 + 전화번호 로그인

전면 무료 운영을 우선할 경우, 참가자 로그인은 `이름 + 휴대전화번호` 검증으로 바꾸는 방안을 검토한다.

권장 플로우:

```text
1. 참가자가 이름과 휴대전화번호를 입력한다.
2. 서버는 전화번호를 숫자만 남기는 방식으로 정규화한다.
3. 등록된 활성 참가자 중 이름 + 휴대전화번호가 정확히 일치하는 사용자를 찾는다.
4. 정확히 1명만 매칭되면 기존 참가자 커스텀 JWT를 발급한다.
5. 0명 또는 2명 이상 매칭되면 로그인시키지 않는다.
6. 실패 응답은 계정 존재 여부를 노출하지 않는 동일 문구를 사용한다.
```

가능하면 이름 + 휴대전화 뒤 4자리보다 이름 + 전체 휴대전화번호를 권장한다. 뒤 4자리는 동명이인이나 번호 충돌 가능성이 커진다. 행사코드 또는 행사 선택값을 추가하면 보안을 조금 더 보완할 수 있다.

보안상 성격:

- OTP는 "해당 연락처를 실제로 소유하고 있음"을 증명한다.
- 이름 + 전화번호 로그인은 "등록 정보를 알고 있음"을 확인한다.
- 이 서비스가 사전 등록된 참가자만 쓰는 폐쇄형 행사 운영 도구라면 MVP 무료 운영용 타협안으로는 실용적이다.

보완 규칙:

- 전체 전화번호 입력 권장
- 전화번호 정규화 비교
- 실패 문구 동일화
- 로그인 시도 rate limit 유지
- 모호한 매칭 차단
- 관리자 1회용 로그인 링크를 현장 예외 수단으로 유지

## 5. 알림 기능은 인증과 분리

중요한 방향은 "인증"과 "안내 발송"을 분리하는 것이다.

현재 OTP 기반 흐름에서는 로그인 자체가 발송 서비스에 의존한다. 이름 + 전화번호 로그인으로 바꾸면 로그인은 자체 처리되고, 알림톡/SMS는 Phase 7에서 선택 기능으로 붙일 수 있다.

무료 운영 모드:

```text
- 외부 발송 API 없음
- OTP 자동 발송 없음
- 참가자는 이름 + 휴대전화번호로 로그인
- 알림/안내는 운영자가 수동으로 처리
- 시스템은 안내문, 대상자 목록, 링크, CSV를 만들어주는 보조 도구 역할
```

유료/확장 운영 모드:

```text
- 기본 로그인은 이름 + 전화번호 또는 OTP 중 정책 선택
- 예약 리마인드, 행사 안내, 미예약 독려를 알림톡으로 발송
- 카카오 미수신/실패 시 SMS fallback
- 발송 로그, 실패 재시도, 중복 방지 키, 백오프 구현
```

## 6. 예상 작업량

이름 + 전화번호 로그인으로 바꾸는 작업은 "큰 재설계"보다는 "인증 진입 방식 교체"에 가깝다. 기존 참가자 커스텀 JWT, RLS, `participantClient`, 역할별 리다이렉션은 재사용 가능하다.

예상 변경 범위:

- 로그인 UI: OTP 2단계 화면을 이름 + 전화번호 입력 화면으로 변경
- 인증 스키마: 이름/전화번호 검증 추가
- 서버 검증: 이름 + 전화번호 매칭 RPC 또는 Edge Function 추가/교체
- JWT 발급: 기존 `participant-otp-verify`의 JWT 서명 흐름 재사용 가능
- 관리자 화면: `최근 OTP 상태`, `인증 가능 채널` 표현을 `로그인 정보`, `최근 로그인`, `현장 링크` 중심으로 재정리
- 문서: `page_auth_layout.md`, `security_transactions.md`, `user_guide.md`, `development_status.md` 갱신 필요
- 테스트: 로그인 검증, 모호한 매칭, 실패 generic 응답, rate limit 테스트 수정

작업량 판단:

- 이메일/SMS/알림톡 OTP를 모두 걷어내고 이름 + 전화번호 로그인으로 단순화: 중간 이하
- Supabase Auth 내장 OTP로 갈아타기: 큼
- 이름 + 전화번호 로그인 유지 + Phase 7 알림 API 대기 구조: 중간

## 7. 현재 판단

현재 단계에서는 전면 무료 운영을 목표로 한다면 다음 방향이 가장 현실적이다.

```text
1. 참가자 로그인은 이름 + 전체 휴대전화번호로 변경한다.
2. OTP 자동 발송은 보류하거나 유료 운영 모드 기능으로 격리한다.
3. 알림톡/SMS는 Phase 7에서 인증이 아닌 운영 알림 기능으로 붙인다.
4. 관리자 1회용 로그인 링크는 현장 예외 대응 수단으로 유지한다.
5. 문서에는 무료 운영 모드와 유료 알림 확장 모드를 명확히 분리한다.
```

아직 확정 전이므로, 실제 구현 전에 아래 결정을 내려야 한다.

- 이름 + 전화번호만 쓸지, 행사코드/행사 선택을 추가할지
- 기존 OTP 테이블과 Edge Function을 보존할지, 비활성화 후 제거할지
- 관리자 화면의 OTP 관련 컬럼을 어떤 운영 표현으로 바꿀지
- Phase 7을 "알림 서비스 연동"으로만 남길지, "OTP 재도입 옵션"까지 포함할지
