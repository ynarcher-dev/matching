# UI 피드백 메시지 전수조사 및 개선안

작성일: 2026-07-01  
상태: 문서 작업 전용 초안  
범위: 버튼 클릭, 폼 제출, 파일/사진 업로드, 예약/배정/권한/알림 설정 등 사용자 액션 후 화면에 표시되는 Toast, Alert, 인라인 상태 문구

> 이 문서는 액션 피드백을 어떻게 보여줄지 정리하기 위한 UX 판단 문서다. 데이터 구조나 RPC 동작 변경은 다루지 않는다.

## 1. 목적

현재 시스템은 버튼을 눌렀을 때 결과를 화면 내부에 표시하는 방식이 섞여 있다.

- 일부는 전역 Toast로 표시된다.
- 일부는 카드나 모달 내부 Alert로 표시된다.
- 일부는 버튼 텍스트 또는 인라인 상태 문구로 표시된다.
- 일부 성공 액션은 아무 문구 없이 모달을 닫거나 화면만 갱신된다.

사용자 선호와 운영 UX를 기준으로 보면, 대부분의 클릭 액션 결과는 Toast가 더 적합하다. 반대로 항상 박혀 있는 Alert는 실제로 사용자가 지금 알아야 하는 지속 상태일 때만 남기는 것이 좋다.

이 문서는 다음을 목표로 한다.

- 현재 액션 피드백 문구와 위치를 정리한다.
- Toast, Alert, 인라인 상태, 모달 본문을 어떤 기준으로 쓸지 정한다.
- 어지럽게 상시 배치된 안내를 유지/변경/삭제 대상으로 나눈다.
- 주요 문구를 before/after 형태로 제안한다.

## 2. 현재 구조 요약

### 2.1 전역 Toast

구현:

- `src/stores/toastStore.ts`
- `src/components/common/ToastViewport.tsx`

현재 사용:

- `src/components/expert/CounselingLogForm.tsx`의 상담일지 임시저장, 제출, 제출 취소에만 사용된다.

현재 문구:

| 상황 | 문구 |
| :--- | :--- |
| 임시저장 성공 | `임시저장되었습니다` |
| 임시저장 실패 | `임시저장에 실패했습니다` |
| 상담일지 제출 성공 | `상담일지가 제출되었습니다` |
| 제출 실패 | `제출에 실패했습니다` |
| 제출 취소 성공 | `제출이 취소되었습니다` |
| 제출 취소 실패 | `제출 취소에 실패했습니다` |

평가:

- 클릭 액션 결과를 알려주는 용도로 가장 적합하다.
- 사용 범위가 너무 좁다.
- 현재 성공/실패 문구가 짧아서 좋지만, 일부는 다음 상태를 함께 말하면 더 명확하다.

### 2.2 내부 Alert

구현:

- `src/components/common/Alert.tsx`

현재 사용:

- 로그인 실패
- 데이터 로딩 실패
- 설정 저장 성공/실패
- 테스트 발송 성공/실패
- 슬롯 생성 성공/실패
- AI 제안 생성 결과
- 파일/사진 업로드 결과
- 권한/배정/예약/취소 실패
- 진행 단계 안내
- 편집 잠금 사유
- 위험 경고

평가:

- 에러, 위험 경고, 지속 상태에는 적합하다.
- 단순 클릭 성공까지 Alert로 남으면 화면이 누적 피드백으로 복잡해진다.
- 같은 화면 안에서 안내 Alert, 성공 Alert, 실패 Alert, 빈 상태 문구가 동시에 보이는 경우가 있어 시각적 우선순위가 흐려진다.

### 2.3 인라인 상태 문구

대표 구현:

- `src/components/common/SaveStatus.tsx`
- `src/components/expert/CounselingLogForm.tsx`의 `AutosaveIndicator`
- `src/components/admin/participantCells.tsx`의 파일 열기 버튼 상태
- `src/components/common/FileUploadField.tsx`
- `src/components/common/PhotoPicker.tsx`

평가:

- 자동저장, 파일 선택, 업로드 대기, 복사 완료처럼 현재 컴포넌트 상태를 보여주는 데 적합하다.
- 버튼 클릭 결과를 오래 남기는 용도로 쓰면 화면이 지저분해진다.

## 3. 권장 표시 원칙

### 3.1 Toast로 보낼 것

클릭 액션의 성공/실패 결과는 기본적으로 Toast로 보낸다.

대상:

- 저장, 생성, 수정, 삭제
- 예약, 변경, 취소
- 권한 부여, 회수, 세션 무효화
- 테스트 발송
- 재시도
- 파일 보기 실패
- 사진 업로드 완료
- 링크 복사
- CSV 등록 완료
- AI 제안 생성/확정 완료
- 슬롯 생성/초기화 완료

원칙:

- 성공 Toast는 짧게 쓴다.
- 실패 Toast는 사용자가 복구할 수 있게 쓴다.
- 서버/RPC 에러가 한국어 사용자 문구면 그대로 쓰되, 앞에 맥락을 붙인다.
- 같은 액션의 성공 Alert는 제거한다.

예:

- `저장되었습니다.`
- `예약이 완료되었습니다.`
- `예약을 취소했습니다.`
- `사진 3장을 업로드했습니다.`
- `권한을 회수했습니다.`
- `처리하지 못했습니다. 잠시 후 다시 시도해 주세요.`

### 3.2 Alert로 유지할 것

화면에 계속 남아 있어야 하는 상태는 Alert로 유지한다.

대상:

- 권한 없음 또는 편집 불가
- 행사 상태 때문에 액션이 불가능함
- 위험한 설정 경고
- 데이터 로딩 실패
- 폼 검증 오류 중 사용자가 해당 위치에서 고쳐야 하는 것
- 제출 완료 후 수정 불가 같은 지속 상태
- 현재 발송 모드, 전역 발송 비활성 등 화면 맥락

원칙:

- Alert는 한 화면에서 최대 1~2개가 적당하다.
- 같은 레벨의 Alert가 여러 개 쌓이면 `StatusPanel`이나 `ValidationSummary`로 묶는다.
- 성공 Alert는 지속 상태일 때만 쓴다.

### 3.3 인라인 상태로 유지할 것

사용자의 입력 또는 현재 필드 상태와 밀접한 것은 인라인으로 둔다.

대상:

- 자동저장 중/저장됨/저장 실패
- 파일 선택됨
- 업로드 대기 n장
- 복사됨 버튼 상태
- 필드 단위 검증 오류
- 특정 row의 파일 열기 중/실패

원칙:

- 입력 필드 바로 아래 또는 버튼 안에 표시한다.
- 페이지 상단 Alert로 끌어올리지 않는다.

### 3.4 모달 본문으로 유지할 것

액션 전에 사용자가 판단해야 하는 내용은 모달 본문에 남긴다.

대상:

- 예약 확인
- 예약 취소 확인
- 행사 취소 경고
- 상태 수동 변경 경고
- 테이블 삭제 확인
- 참가자 제외 시 배치 해제 안내
- 권한 회수 확인
- 노쇼 처리 사유 입력

원칙:

- 확인 전 경고는 모달 본문.
- 확인 후 성공/실패는 Toast.
- 사유 누락은 모달 내부 인라인 오류.

## 4. Before / After 제안

### 4.1 상담일지

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 임시저장 성공 | `임시저장되었습니다` | `임시저장했습니다.` | Toast | 유지, 마침표 포함 |
| 임시저장 실패 | `임시저장에 실패했습니다` | `임시저장하지 못했습니다. 다시 시도해 주세요.` | Toast | 변경 |
| 최종 제출 성공 | `상담일지가 제출되었습니다` | `상담일지를 제출했습니다.` | Toast | 변경 |
| 제출 실패 | `제출에 실패했습니다` | `상담일지를 제출하지 못했습니다. 필수 항목과 네트워크 상태를 확인해 주세요.` | Toast + 폼 오류는 Alert | 변경 |
| 제출 취소 성공 | `제출이 취소되었습니다` | `제출을 취소했습니다. 다시 편집할 수 있습니다.` | Toast | 변경 |
| 자동저장 비활성 | `자동저장 비활성 (대기/진행 세션만)` | `대기/진행 세션에서만 자동저장됩니다.` | 인라인 | 유지, 문구 완화 |
| 저장 실패 | `저장 실패: {error}` | `자동저장 실패: {error}` | 인라인 | 유지 |

권장:

- 상담일지는 현재 Toast 방향이 맞다.
- 완료된 상담일지 안내 Alert는 유지한다. 제출 완료 상태 자체가 지속 상태이기 때문이다.
- 문항 로딩 실패 Alert도 유지한다.

### 4.2 스타트업 예약

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 예약 확인 | `해당 시간[...]에 ... 전문가와의 상담을 예약하시겠습니까?` | 동일하되 시간/전문가를 줄바꿈 카드형으로 정리 | ConfirmModal | 유지 |
| 예약 성공 | 없음, 모달 닫힘 | `상담 예약이 완료되었습니다.` | Toast | 추가 |
| 예약 실패 | 모달 내부 RPC 에러 | `예약하지 못했습니다. {error}` | Toast + 모달 유지 | 변경 |
| 예약 취소 확인 | `상담 일정을 취소하시겠습니까?...` | `상담 일정을 취소할까요? 취소한 시간은 다른 기업이 바로 예약할 수 있습니다.` | ConfirmModal | 문구 변경 |
| 예약 취소 성공 | 없음, 모달 닫힘 | `예약을 취소했습니다.` | Toast | 추가 |
| 예약 변경 실패 | 모달 내부 RPC 에러 | `예약 시간을 변경하지 못했습니다. {error}` | Toast + 모달 유지 | 변경 |
| 변경 후보 없음 | `변경 가능한 빈 슬롯이 없습니다...` | `변경 가능한 시간이 없습니다.` | 모달 내부 EmptyState | 변경 |

권장:

- 예약/변경/취소의 결과는 Toast로 통일한다.
- ConfirmModal 내부 error Alert는 중복이 될 수 있으므로, 실패 Toast를 띄우고 모달은 그대로 열어 두는 방식이 더 깔끔하다.
- 다만 실패 원인이 폼 안 선택값 문제라면 모달 내부에 남긴다.

- 빈 상태 안내는 Alert보다 EmptyState에 가깝게 가볍게 둔다.

### 4.4 만족도 조사

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 제출 성공 후 | `만족도 조사를 제출해 주셔서 감사합니다. 제출한 응답은 수정할 수 없습니다.` | `제출한 응답은 수정할 수 없습니다.` | 지속 Alert 또는 Summary 상태 | 유지, 축약 |
| 제출 성공 순간 | 없음 | `만족도 조사를 제출했습니다.` | Toast | 추가 |
| 필수 누락 | `필수 항목입니다: {title}` | `{title} 문항에 응답해 주세요.` | 인라인/Alert | 변경 |
| 단일 선택 오류 | `{title}: 하나만 선택해 주세요.` | `{title} 문항은 하나만 선택할 수 있습니다.` | 인라인/Alert | 변경 |
| 서버 제출 실패 | 서버 에러 | `만족도 조사를 제출하지 못했습니다. {error}` | Toast 또는 폼 Alert | 변경 |

권장:

- 제출 성공 순간은 Toast.
- 제출 완료 후 읽기 전용 상태는 Alert로 유지해도 된다.
- 검증 오류는 Toast보다 문항 근처 인라인 또는 폼 하단 Alert가 좋다.

### 4.5 관리자 알림 설정

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 전역 설정 저장 성공 | `저장되었습니다.` | `알림 설정을 저장했습니다.` | Toast | Alert 제거 |
| 행사 알림 정책 저장 성공 | `저장되었습니다.` | `행사 알림 정책을 저장했습니다.` | Toast | Alert 제거 |
| 저장 실패 | `{error}` 또는 `저장에 실패했습니다.` | `저장하지 못했습니다. {error}` | Toast + 필드 오류는 인라인 | 변경 |
| 테스트 발송 성공 | `테스트 발송 성공 (provider).` | `테스트 발송을 요청했습니다. ({provider})` | Toast | Alert 제거 |
| 테스트 발송 실패 | `테스트 발송에 실패했습니다.` | `테스트 발송에 실패했습니다. 설정을 확인해 주세요.` | Toast | 변경 |
| 공급사 불완전 | `공급사 설정이 불완전합니다...` | `공급사 설정이 불완전합니다. API 키와 발신번호를 확인해 주세요.` | Alert | 유지 |
| 전역 발송 OFF 안내 | 긴 안내 Alert | 짧은 StatusBanner: `전역 발송 비활성: 외부 API를 호출하지 않습니다.` | 상단 상태 배너 | 변경 |

권장:

- 설정 저장 성공 Alert는 화면에 남을 이유가 약하다. Toast로 바꾼다.
- 전역 발송 모드는 사용자가 현재 화면에서 계속 알아야 하므로 Alert 또는 StatusBanner로 유지한다.
- 위험 경고인 `실제 발송 활성화` 안내는 유지하되, 에러 tone보다 warning tone이 더 적합하다.

### 4.6 슬롯 생성 / AI 배정

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 슬롯 생성 성공 | `슬롯 {n}개를 생성했습니다.` | 동일 | Toast | Alert 제거 |
| 빈 슬롯 초기화 성공 | `빈 슬롯 {n}개를 삭제했습니다.` | 동일 | Toast | Alert 제거 |
| 슬롯 생성 실패 | RPC 에러 Alert | `슬롯을 생성하지 못했습니다. {error}` | Toast 또는 폼 Alert | 변경 |
| AI 제안 생성 성공 | `제안 n건 생성 · 미배치 n건...` | `AI 제안을 생성했습니다.` + 상세는 리포트/요약 카드 | Toast + Summary | 분리 |
| AI 제안 생성 실패 | 에러 Alert | `AI 제안을 생성하지 못했습니다. {error}` | Toast | 변경 |
| AI 확정 결과 | `{applied}건을 매칭 슬롯에 반영했습니다...` | 동일 | 결과 리포트 Alert/Card | 유지 |

권장:

- 단순 성공은 Toast.
- AI 확정 결과처럼 후속 확인이 필요한 결과는 화면 내부 리포트로 유지한다.
- 생성 성공 Alert는 화면 상단을 밀어내므로 Toast와 요약 카드로 나누는 것이 좋다.

### 4.7 진행 현황 / 노쇼 / 대체 매칭 / 사진

| 상황 | Before | After | 표시 방식 | 판단 |
| :--- | :--- | :--- | :--- | :--- |
| 세션 상태 변경 성공 | 없음 | `세션 상태를 변경했습니다.` | Toast | 추가 |
| 세션 상태 변경 실패 | Alert `{error}` | `세션 상태를 변경하지 못했습니다. {error}` | Toast | 변경 |

| 노쇼 처리 실패 | ConfirmModal Alert | `노쇼 처리하지 못했습니다. {error}` | Toast + 모달 유지 | 변경 |
| 대체 매칭 성공 | 모달 닫힘 | `대체 매칭을 완료했습니다.` | Toast | 추가 |
| 대체 매칭 실패 | 모달 Alert | `대체 매칭하지 못했습니다. {error}` | Toast + 모달 유지 | 변경 |

| 사진 삭제 실패 | `삭제에 실패했습니다. 다시 시도해 주세요.` | 동일 | Toast 또는 ConfirmModal Alert | Toast 권장 |

권장:

- 현장 진행 화면은 계속 봐야 하는 데이터가 많으므로 성공 Alert를 남기면 방해된다.
- 사진 업로드 결과도 Toast가 적합하다.
- 사진 업로드 제한, 파일 형식 오류는 선택 영역 인라인으로 유지한다.

## 5. 유지 / 변경 / 제거 분류

### 5.1 유지

유지할 Alert:

- 데이터 로딩 실패: `... 불러오지 못했습니다. 새로고침 후 다시 시도해 주세요.`
- 편집 잠금 사유: 취소/진행/종료/응답 발생으로 편집 불가
- 위험 경고: 행사 취소, 상태 수동 변경, 실제 발송 활성화
- 제출 완료 후 수정 불가 안내
- 민감정보 1회 표시 안내
- CSV 검증 결과
- AI 확정 결과 리포트

이유:

- 사용자가 화면에 머무는 동안 계속 알아야 하는 상태다.
- Toast로 사라지면 중요한 맥락을 놓칠 수 있다.

### 5.2 Toast로 변경

Alert에서 Toast로 옮길 것:

- 설정 저장 성공
- 슬롯 생성/삭제 성공
- 사진 업로드 성공
- 테스트 발송 성공/실패
- 세션 상태 변경 실패
- 예약/취소/변경 실패
- 권한 부여/회수 실패
- 재시도 실패
- 순서 변경 실패

아예 새로 Toast를 추가할 것:

- 예약 성공
- 예약 변경 성공
- 예약 취소 성공
- 참가자 저장/등록 성공
- 운영자 저장/수정 성공
- 권한 부여/회수 성공
- 테이블 추가/삭제 성공
- 문항 추가/수정/삭제 성공
- URL 추가/삭제 성공
- 소개서 업로드/교체/해제 성공
- 세션 무효화 성공

### 5.3 인라인으로 변경

Alert 또는 상단 메시지보다 인라인이 나은 것:

- URL 미입력
- 발급/회수/배정 사유 누락
- 파일 형식/용량 검증
- 분야 최대 개수 초과
- 만족도/상담일지 문항 필수 누락
- 특정 row 파일 열기 실패
- 자동저장 실패

### 5.4 제거 또는 축약

축약할 상시 안내:

- 전역 발송 OFF 안내: 긴 문장 대신 상태 배너
- 소개서 미제출 안내: Alert보다 EmptyState
- 진행 단계 아님 안내: 긴 Alert보다 짧은 상태 배너
- `저장되었습니다.`류 성공 Alert: Toast로 옮긴 뒤 제거

## 6. 문구 톤 가이드

### 6.1 성공

원칙:

- 능동형으로 짧게 쓴다.
- `되었습니다`보다 `했습니다`를 기본으로 쓴다.
- 다음 행동이 있으면 한 문장만 추가한다.

Before:

- `저장되었습니다.`
- `상담일지가 제출되었습니다.`
- `링크가 발급되었습니다.`

After:

- `저장했습니다.`
- `상담일지를 제출했습니다.`
- `링크를 발급했습니다. 지금 전달해 주세요.`

### 6.2 실패

원칙:

- 무엇을 못 했는지 먼저 말한다.
- 복구 방법이 있으면 짧게 붙인다.
- 서버 원문은 뒤에 붙인다.

Before:

- `제출에 실패했습니다`
- `{error}`
- `테스트 발송에 실패했습니다.`

After:

- `상담일지를 제출하지 못했습니다. 필수 항목을 확인해 주세요.`
- `처리하지 못했습니다. {error}`
- `테스트 발송에 실패했습니다. 공급사 설정을 확인해 주세요.`

### 6.3 경고

원칙:

- 사용자가 확정 전에 판단해야 하는 위험을 구체적으로 말한다.
- 에러 tone보다 warning tone을 권장한다.
- 긴 설명은 접을 수 있는 보조 설명 또는 모달 본문에 둔다.

Before:

- `OFF 상태(기본값)에서는 외부 API를 호출하지 않습니다. ON으로 변경하면 행사별 정책에 따라 실제 알림이 발송되며 비용이 발생할 수 있습니다.`

After:

- `실제 발송을 켜면 비용이 발생할 수 있습니다. 행사별 정책에 따라 외부 알림이 발송됩니다.`

### 6.4 빈 상태

원칙:

- Alert로 보이지 않게 한다.
- 사용자가 할 수 있는 다음 행동을 같이 둔다.

Before:

- `아직 제출된 소개서가 없습니다. PDF 파일을 업로드해 주세요.`

After:

- `제출된 소개서가 없습니다.`
- 옆 또는 아래 버튼: `파일 업로드`

## 7. 구현 방향

### 7.1 Toast API 확장

현재:

- `toast.success(message)`
- `toast.error(message)`
- `toast.info(message)`

제안:

- `toast.success(message, options?)`
- `toast.error(message, options?)`
- `toast.info(message, options?)`
- `toast.warning(message, options?)`

필요 옵션:

- `description`
- `durationMs`
- `actionLabel`
- `onAction`

예:

```ts
toast.success('예약이 완료되었습니다.');
toast.error('예약하지 못했습니다.', { description: error.message });
toast.warning('일부 사진 업로드에 실패했습니다.', { description: '실패분은 다시 시도해 주세요.' });
```

### 7.2 Mutation 성공/실패 처리 규칙

현재 많은 mutation은 성공 시 query invalidate만 하고 UI 피드백이 없다.

권장:

- `onSuccess`: Toast + 모달 닫기 + invalidate
- `onError`: Toast + 모달 유지
- 폼 검증 실패: Toast 금지, 폼 내부 인라인/Alert

예:

```ts
mutation.mutate(input, {
  onSuccess: () => {
    toast.success('저장했습니다.');
    onClose();
  },
  onError: (e) => {
    toast.error('저장하지 못했습니다.', { description: (e as Error).message });
  },
});
```

### 7.3 Alert 역할 분리

현재 `Alert`는 error/info/success만 있다.

제안:

- `error`: 데이터 로딩 실패, 처리 불가
- `warning`: 비용 발생, 삭제/취소 위험
- `info`: 현재 상태 안내
- `success`: 지속 성공 상태만 제한적으로 사용

추가 컴포넌트 후보:

- `StatusBanner`: 페이지/패널 상단의 지속 상태
- `ValidationSummary`: 폼 제출 전 오류 목록
- `EmptyState`: 빈 목록/미제출/후보 없음
- `InlineFeedback`: 필드나 row 바로 아래 상태

## 8. 우선순위

### P0: 공통 규칙 확정

- Toast와 Alert 사용 기준 확정
- 성공 Alert는 Toast로 이동한다는 원칙 확정
- 위험 경고는 warning tone 도입
- 빈 상태는 EmptyState로 분리

### P1: 클릭 액션 Toast화

- 예약/변경/취소
- 설정 저장
- 슬롯 생성/초기화
- 사진 업로드/삭제
- 권한 부여/회수
- 참가자/운영자 저장
- 문항 저장/삭제

### P2: 상시 Alert 정리

- 전역/행사 알림 설정의 긴 안내 축약
- 소개서 미제출, 후보 없음, 등록 없음 안내를 EmptyState로 변경
- 진행 단계 안내를 StatusBanner로 정리

### P3: 문구 정리

- `되었습니다`를 `했습니다` 중심으로 통일
- `URL 을` → `URL을`처럼 띄어쓰기 정리
- 서버 에러 원문 앞에 사용자 맥락 추가
- 문항 검증 오류를 더 자연스럽게 변경

## 9. 결론

현재 UI 피드백은 기능적으로는 대부분 존재하지만, 표시 위치가 섞여 있어 화면이 복잡하게 느껴진다. 가장 큰 개선 방향은 명확하다.

1. 버튼 클릭 결과는 Toast로 보낸다.
2. 화면에 계속 남아야 하는 상태만 Alert로 유지한다.
3. 입력값 문제는 해당 입력 근처 인라인으로 보여준다.
4. 빈 상태는 Alert가 아니라 EmptyState로 보여준다.
5. 위험 경고는 모달 본문 또는 warning banner로 보여준다.

이 기준을 적용하면 사용자는 액션 결과를 빠르게 확인하면서도, 각 화면에는 정말 중요한 지속 상태만 남게 된다.
