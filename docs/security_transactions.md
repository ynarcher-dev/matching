# 인증, 권한 및 트랜잭션 정책

본 문서는 화면 명세와 [DB 스키마](./db_schema.md)에 공통 적용되는 보안 및 서버 처리 원칙을 정의합니다.

## 1. 인증

- 관리자와 현장 스태프는 Supabase Auth 이메일·비밀번호를 사용하며 관리자 MFA 적용을 권장합니다.
- 전문가와 스타트업은 등록된 이메일 또는 휴대전화 번호로 6자리 OTP를 요청하고 검증하여 로그인합니다.
- OTP 원문은 저장하거나 로그에 남기지 않으며, DB에는 단방향 해시와 만료·사용·시도 정보를 저장합니다.
- OTP는 기본 5분 만료, 재요청 60초 제한, 챌린지당 최대 5회 검증 실패 제한을 적용합니다.
- OTP 요청 API는 계정 존재 여부를 노출하지 않는 동일한 일반 응답을 반환합니다.
- 하나의 입력값이 여러 활성 참가자와 모호하게 일치하면 누구에게도 OTP를 발송하지 않고 동일한 일반 응답을 반환하며, 운영본부 확인 경로를 안내합니다.
- 새 OTP 발급 시 같은 목적의 이전 미사용 OTP를 무효화하며, 검증 성공 시 해당 OTP를 원자적으로 사용 처리합니다.
- 참가자 세션은 기본 12시간 또는 행사 종료 시각 중 빠른 시점에 만료합니다. 관리자는 `session_version` 증가로 기존 세션을 무효화할 수 있습니다.
- 공통 행사코드는 단독 인증 수단으로 사용할 수 없습니다.
- 로그인은 서버 함수 또는 인증 RPC에서 검증하며 브라우저가 사용자 해시를 직접 조회할 수 없습니다.

## 2. 권한과 RLS

- `ADMIN`: 모든 행사 운영 데이터 관리. 엑셀 다운로드는 `ADMIN`(일반 관리자) 이상 계정에서 모두 가능합니다(MFA 권장). 상태 강제 변경은 `is_super_admin = true`인 계정만 가능.
- `STAFF`: 배정된 현장 업무에서 스타트업 출석 처리. 최고 관리자 기능과 엑셀 다운로드 불가.
- `EXPERT`: 본인의 슬롯, 매칭된 스타트업 공개 프로필과 소개서, 본인이 작성한 상담일지만 접근.
- `STARTUP`: 본인의 행사 참가 정보, 예약, 공개된 전문가 정보와 공개 허용된 상담 코멘트만 접근.
- OTP 해시와 챌린지 정보, 관리자 전용 평가 점수, 다른 참가자의 개인정보는 클라이언트 쿼리에서 노출하지 않습니다.
- Supabase Storage의 소개서와 프로필 사진은 공개 URL 대신 권한과 만료 시간이 적용된 Signed URL을 사용합니다.

## 3. RPC 전용 변경 작업

다음 작업은 테이블 직접 변경을 허용하지 않고 DB RPC 또는 서버 함수로만 처리합니다.

- 참가자 OTP 요청·검증, 세션 무효화, 1회용 로그인 링크 발급
- 예약 신청, 변경, 취소
- 관리자 강제 배정
- 자동 배치 제안 확정
- 상담 시작, 노쇼 및 세션 취소
- 상담일지 제출과 세션 완료
- 출석 체크
- 행사 상태 수동 변경

각 함수는 호출자의 역할과 행사 참가 여부를 확인하고, 필요한 변경과 이력 기록을 하나의 트랜잭션으로 처리합니다.

## 4. 알림 처리

- 예약과 상담 트랜잭션을 먼저 완료한 후 알림을 비동기로 처리합니다.
- 이벤트와 수신자 조합의 `idempotency_key`로 중복 발송을 방지합니다.
- 실패 시 지수 백오프로 최대 3회 재시도하고 영구 실패는 관리자 화면에 표시합니다.
- 초기 개발 환경은 Mock 서비스, 운영 환경은 Solapi 어댑터를 사용합니다.

## 5. 출석 처리 (대시보드 / QR)

- 기본 출석 수단은 대시보드 화면의 원클릭 체크 버튼이며, QR 스캔은 보조 수단으로 지원합니다.
- QR에는 사용자 UUID를 직접 넣지 않고 행사·사용자·만료 시각이 서명된 30초 만료 토큰을 사용합니다.
- 서버 검증을 통과한 토큰만 출석 처리에 사용할 수 있습니다.
- 전문가는 본인 출석을 처리합니다. 스타트업 출석은 관리자 또는 스태프가 처리하는 것이 기본이나, 전문가도 본인 대시보드 세션 카드에서 해당 스타트업의 출석 여부를 수동으로 확인·체크할 수 있습니다.
- QR 사용이 어려운 경우 이름 또는 기업명 검색을 통한 수동 체크인을 허용하고 처리 이력을 남깁니다.
