# 개발용 시드 데이터 컨벤션 (마이그레이션 후 데이터 주입)

> 작성 2026-06-26. 목적: **마이그레이션을 적용한 뒤에는 화면에서 바로 확인할 수 있도록
> 더미 데이터를 함께 넣는다**는 작업 규칙을 고정한다.

---

## 1. 규칙 (Rule)

**스키마/기능 마이그레이션을 적용하면, 그 직후 해당 변경으로 새로 생긴 구조가
관리자 화면에서 눈으로 확인 가능하도록 더미 데이터를 채워 넣는다.**

- "코드/마이그레이션이 들어갔다"로 끝내지 않고, **화면에 데이터가 보이는 상태**까지를 한 세트로 본다.
- 데이터가 없으면 사용자가 "만들어진 건 맞는데 어떻게 보이는지 판단이 안 된다"는 상황이 생긴다 → 이를 방지.

## 2. 언제 넣고, 언제 생략하나 (Scope)

| 마이그레이션 성격 | 시드 주입 |
|---|---|
| 새 테이블 / 컬럼 / 사용자에게 **보이는 구조** 추가 | ✅ 넣는다 (그 구조를 채우는 행 포함) |
| 기존 화면에 새 데이터 흐름이 생김 (예: 제안·배치 결과) | ✅ 넣는다 (확인용 결과 행 포함) |
| 순수 RPC 동작/보안 가드 수정 등 **새 데이터가 필요 없는 변경** | ⬜ 생략 가능 (기존 시드로 확인) |

판단 기준: **"이 마이그레이션으로 새로 생긴 걸 사용자가 화면에서 보려면 무슨 행이 필요한가?"**
필요한 행이 있으면 넣고, 기존 시드로 충분하면 생략한다.

## 3. 적용 방법 (How) — `0016_dev_seed.sql` 패턴 그대로

이 프로젝트는 원격 DB에 `supabase db push`(자격증명 저장됨, 무프롬프트)로만 SQL을 적용한다.
따라서 시드도 **번호 마이그레이션 파일**로 작성해 push 한다. 최초 기준 구현은
[`supabase/migrations/0016_dev_seed.sql`](../supabase/migrations/0016_dev_seed.sql).

작성 시 **반드시 지킬 4가지**:

1. **고정 UUID** — 재참조·재실행을 위해 결정적 UUID 사용
   (예: 전문가 `e0000000-…-0000000000NN`, 스타트업 `50000000-…-0000000000NN`, 행사 `a0000000-…-000N`).
2. **단일 가드로 idempotent** — 대표 행(예: 기준 행사) 존재 시 전체 스킵.
   `IF EXISTS (SELECT 1 FROM … WHERE id = …) THEN RAISE NOTICE '이미 적용됨'; RETURN; END IF;`
3. **RPC 우회** — 관리자 가드(`current_app_role() = 'ADMIN'`)가 있는 RPC는 마이그레이션 컨텍스트에서
   막히므로, 슬롯/예약/제안 등은 **RPC 호출 대신 직접 INSERT**한다. (마이그레이션은 테이블 owner =
   RLS·SECURITY DEFINER 가드 우회)
4. **제약/트리거 준수** — CHECK·FK·분야 최대 3개 트리거([`0013`](../supabase/migrations/0013_fields_limit.sql)) 등을
   만족하는 값만 넣는다.

그리고 **롤백 스니펫**(이 시드만 일괄 삭제하는 DELETE)을 파일 하단 주석으로 남긴다.

## 4. 마이그레이션 번호 주의

시드도 번호를 **선점**한다. 다른 작업이 특정 번호를 예약해 뒀다면 충돌하지 않도록
**다음 빈 번호**를 쓰고, 핸드오프 메모(NEXT)에 어긋난 예약이 있으면 갱신한다.
> 예: 0016은 원래 보안픽스용으로 거론됐으나 더미 시드가 선점 → 보안픽스는 0017로 이동.

## 5. 정리 (Cleanup)

더미 데이터는 운영 데이터가 아니다. 실테스트/배포 전 정리가 필요하면
각 시드 마이그레이션 하단의 **롤백 스니펫**을 수동 실행한다(행사·고정 UUID 유저 DELETE → CASCADE).
현재 적용된 시드 현황은 메모리 `[[dev-seed-data]]` 참고.

### 한 줄 요약

번호 시드 마이그레이션으로 함께 push 한다.**
