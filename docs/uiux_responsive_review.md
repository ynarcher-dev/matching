# UI/UX and Responsive Review

이 문서는 현재 구현된 비즈니스 매칭 서비스의 UI/UX를 운영 관점에서 점검하고, 특히 반응형 UI 기준을 정리하기 위한 리뷰 문서다. 핵심 전제는 "모든 기능을 모바일 앱처럼 만들 필요는 없지만, 현장 대응에 쓰이는 화면은 모바일 친화적이어야 한다"는 것이다.

## 1. 판단 기준

### 1.1 화면 성격 구분

서비스 화면은 크게 두 종류로 나누어 평가한다.

| 분류 | 대표 화면 | 우선 UX |
| :--- | :--- | :--- |
| 백오피스 운영 화면 | 행사 목록, 참가자 DB, 설문/상담 문항 빌더, 리포트, AI 배치 검토 | 데이터 테이블, 검색, 필터, 정렬, 페이지네이션 |
| 현장 대응 화면 | 스타트업 예약, 전문가 대시보드, 진행 현황, 출석/노쇼/상담 시작, 향후 staff 화면 | 모바일 가독성, 큰 터치 타깃, 현재 해야 할 액션 우선 |

백오피스 화면은 데스크톱 생산성이 중요하다. 반대로 현장 대응 화면은 사용자가 노트북보다 휴대폰을 들고 움직일 가능성이 높으므로 모바일 사용성을 우선한다.

### 1.2 공통 점검 항목

- 대량 데이터를 다루는 화면은 카드 나열보다 테이블을 우선한다.
- 테이블에는 검색, 필터, 정렬, 페이지네이션이 필요하다.
- 검색은 단순 키워드뿐 아니라 운영자가 실제로 찾는 조건을 반영해야 한다.
- 모바일 현장 화면은 한 손 조작을 고려해 주요 액션을 큰 버튼으로 제공한다.
- 상태 색상, 배지, 카드/패널 라운드 값은 화면별 임의 조합이 아니라 공통 규칙으로 관리한다.
- 표/그리드가 필요한 경우 모바일에서는 가로 스크롤만 제공하는 데서 끝내지 않고, 필요하면 모바일 전용 리스트/현재 시간대 보기로 보완한다.

### 1.3 데이터 테이블 공통 기준

백오피스의 데이터 테이블형 DB와 결과/로그 화면은 다음 기준을 기본으로 한다.

- 기본 페이지 크기는 30개다.
- 검색, 필터, 정렬, 페이지네이션을 함께 제공한다.
- 검색은 이름/제목뿐 아니라 기업명, 소속, 연락처, 상태, 기간 등 운영자가 실제로 찾는 조건을 포함한다.
- 데이터가 많아지는 화면은 프론트 전체 로딩 후 필터링보다 서버 쿼리 기반 페이지네이션을 우선 검토한다.
- 카드형 목록은 현장 조작이나 요약 확인에 적합한 화면에 한정하고, 대량 관리 화면은 테이블을 기본으로 한다.

## 2. 현재 구현 평가

### 2.1 잘 되어 있는 점

- 공통 레이아웃은 데스크톱 고정 사이드바와 모바일 드로어 구조를 갖추고 있다.
- 참가자 DB 관리는 카드가 아니라 테이블로 구현되어 있어 대량 데이터 화면의 방향이 좋다.
- 스타트업 예약 화면과 진행 현황 화면은 시간표/매트릭스 구조를 사용해 예약 상태를 한눈에 볼 수 있다.
- 전문가 대시보드는 프로필, 행사 선택, 현재 세션, 일정 리스트가 카드 흐름으로 구성되어 모바일 화면에 비교적 잘 맞는다.
- 모바일에서 표가 필요한 화면은 가로 스크롤 컨테이너를 사용해 레이아웃 붕괴를 피하고 있다.

### 2.2 아쉬운 점

- 행사 목록은 검색과 상태 필터가 있지만 카드 그리드 중심이라 행사가 많아질수록 비교와 탐색이 어렵다.
- 행사 목록, 참가자 DB, 알림 로그, 상담 이력 등 목록성 화면에 페이지네이션이 없다.
- 일부 검색은 제목/이름/이메일 정도에 머물러 있어 운영 조건 검색이 부족하다.
- 알림 로그와 상담 이력처럼 누적되는 데이터는 현재 리스트/카드 흐름이어서 장기 운영 시 탐색성이 떨어질 수 있다.
- 행사 상세의 참가자 지정 패널은 후보 검색은 있으나 현재 지정 목록을 대량으로 관리하기에는 테이블/필터/정렬이 부족하다.
- 공통 `Card`는 `rounded-2xl`이지만 내부 패널, 테이블 컨테이너, 버튼, 배지에는 `rounded-xl`, `rounded-lg`, `rounded-md`, `rounded-full`이 혼재한다.
- 상태 색상에 `brand`, `surface`, `border` 토큰 외에도 emerald, violet, orange, amber, blue, red 계열이 직접 사용되어 화면 간 색상 규칙이 흐려질 수 있다.

## 3. 화면별 개선 방향

### 3.1 행사 목록

현재 방향: 카드 그리드 + 상태 필터 + 행사명 검색

권장 방향:

- 카드 그리드를 기본 테이블로 전환한다.
- 컬럼 예시: 행사명, 상태, 예약 기간, 행사 기간, 스타트업 수, 전문가 수, 자율 예약, 생성일, 주요 액션.
- 상태, 기간, 자율 예약 여부, 취소 여부 필터를 제공한다.
- 서버 페이지네이션 또는 최소한 클라이언트 페이지네이션을 도입한다.
- 모바일에서는 테이블 전체를 억지로 최적화하기보다 간결한 행 카드 또는 필수 컬럼만 남긴 리스트를 제공한다.

### 3.2 참가자 DB 관리

현재 방향: 역할 탭 + 키워드 검색 + 테이블

권장 방향:

- 현재 테이블 방향은 유지한다.
- 페이지네이션, 정렬, 컬럼별 필터를 추가한다.
- 필터 예시: 역할, 분야, 인증 채널 유무, 최근 OTP 상태, 파일 첨부 유무, 기업명/소속, 등록일.
- 많은 데이터를 가져온 뒤 프론트에서만 필터링하는 구조는 장기적으로 서버 쿼리 기반으로 전환한다.
- 모바일은 필수 화면이 아니므로 가로 스크롤 테이블 유지가 허용된다.

### 3.3 행사 상세 - 참가자 지정

현재 방향: 후보 검색 + 칩 다중 선택 + 현재 지정 목록

권장 방향:

- 후보 추가 영역은 검색/다중 선택을 유지하되, 대량 후보에서는 체크박스 테이블 또는 가상 리스트를 고려한다.
- 현재 지정 목록은 테이블화한다.
- 전문가 목록에는 기본 테이블, 분야, 소속, 지정/해제 액션을 함께 보여준다.
- 모바일에서는 현재 지정 목록보다 "추가/해제" 액션을 우선하고, 상세 정보는 접힘 영역으로 둔다.

### 3.4 행사 상세 - 진행 현황

현재 방향: 실시간 타임그리드 + 출석 상태 제어

권장 방향:

- 데스크톱에서는 현재 매트릭스가 유효하다.
- 모바일 현장 대응용으로 별도 "현장 모드"를 제공하는 것이 좋다.
- 현장 모드 예시:
  - 현재 시간대 또는 다음 시간대 세션을 상단에 크게 표시
  - 기업명, 전문가명, 테이블, 시간, 상태를 1개 카드에 압축
  - 출석, 불참, 미정, 상담 시작, 노쇼 처리 버튼을 큰 터치 타깃으로 배치
  - 완료/진행/대기 필터 제공
- 모바일에서는 전체 타임그리드보다 "지금 처리할 항목"을 우선한다.

### 3.5 강제 조정

현재 방향: 전체 슬롯 테이블 + 강제 배정/취소

권장 방향:

- 데스크톱에서는 테이블 유지가 적합하다.
- 모바일 현장에서는 전체 슬롯 조정보다 예외 처리 중심 화면이 필요하다.
- 필터 예시: 빈 슬롯, 충돌 슬롯, 노쇼, 특정 기업, 특정 전문가, 현재 시간대.
- 강제 배정 모달은 모바일에서 선택 항목이 많아질 수 있으므로 검색과 최근/추천 후보를 우선 배치한다.

### 3.6 알림 로그

현재 방향: 요약 카드 + 로그 리스트

권장 방향:

- 누적 로그는 테이블화한다.
- 컬럼 예시: 상태, 채널, 유형, 대상, 생성 시각, 재시도 횟수, 다음 재시도, 오류, 액션.
- 상태/채널/실패 여부/발송 유형 필터를 제공한다.
- 실패 로그만 빠르게 보는 프리셋 필터가 필요하다.

### 3.7 스타트업 예약 화면

현재 방향: 행사 선택, 내 예약, 예약 일정표, 완료 후 만족도/코멘트

권장 방향:

- 모바일 친화가 필요한 핵심 화면이다.
- 모바일 기본 탭은 "시간대별 보기"를 우선 고려한다.
- 전문가별 매트릭스는 정보량은 좋지만 휴대폰에서는 넓어지므로 보조 보기로 둔다.
- 내 예약 영역은 화면 상단에서 유지해 사용자가 현재 예약 상태를 즉시 알 수 있어야 한다.
- 예약 가능/마감/내 예약 상태는 색상뿐 아니라 텍스트와 버튼 상태로도 구분한다.

### 3.8 전문가 대시보드

현재 방향: 전문가 정보, 행사 선택, 현재 세션 카드, 일정 리스트, 상담일지 모달

권장 방향:

- 모바일 현장 화면으로 현재 방향이 좋다.
- 현재 세션 카드가 가장 중요한 정보이므로 진행 중에는 최상단에서 강하게 보여준다.
- 출석 처리, 상담 시작, 상담일지 작성 버튼은 충분히 크게 유지한다.
- 일정 리스트는 "현재/다음/완료" 상태 구분을 강화한다.
- 상담일지 모달은 모바일 키보드가 올라와도 저장/제출 버튼이 접근 가능해야 한다.

### 3.9 Staff 현장 화면

현재 상태: 일부 staff 라우트는 placeholder 상태

권장 방향:

- staff 화면은 모바일 퍼스트로 설계한다.
- 필요한 기능 예시:
  - 기업/참가자 빠른 검색
  - 현재 시간대 체크인 대상 목록
  - 출석/불참 처리
  - 기업별 사진 업로드
  - 업로드 누락/출석 누락 필터
- 카드형 리스트와 큰 액션 버튼을 기본으로 한다.
- 데스크톱 테이블보다 현장 조작 속도를 우선한다.

## 4. 디자인 시스템 정리 방향

### 4.1 라운드 규칙

권장 기본값:

| 요소 | 권장 라운드 |
| :--- | :--- |
| 페이지 섹션/카드 | `rounded-xl` 또는 `rounded-lg` 중 하나로 통일 |
| 테이블 컨테이너 | 카드와 같은 라운드 또는 한 단계 작은 라운드 |
| 입력 필드/셀렉트 | `rounded-lg` |
| 작은 액션 버튼 | `rounded-md` |
| 상태 배지/필터 칩 | `rounded-full` |

현재 `Card`의 `rounded-2xl`은 다소 부드러운 인상이 강하다. 운영형 SaaS 화면이라면 `rounded-xl` 또는 `rounded-lg`로 낮추는 것이 더 차분하고 밀도 있는 인상을 줄 수 있다.

### 4.2 상태 색상 규칙

권장 방향:

- 브랜드 주요 액션: `brand`
- 일반 배경/비활성: `surface`, `muted`, `border`
- 성공/예약 완료/출석: green 계열
- 주의/대기/재시도: amber 계열
- 위험/실패/취소/불참: red 또는 brand danger 계열
- 진행 중/정보: blue 또는 info 계열
- AI/자동 배치: violet 계열을 허용하되, 의미를 문서화한다.

색상은 직접 Tailwind 색상을 화면마다 쓰기보다 `StatusBadge`, `ToneBadge`, `DataPill` 같은 공통 컴포넌트로 감싼다.

### 4.3 공통 컴포넌트 후보

- `DataTable`: 정렬, 빈 상태, 로딩 상태, 가로 스크롤, 모바일 대체 렌더링 옵션
- `Pagination`: 페이지 크기, 현재 범위, 이전/다음
- `FilterBar`: 검색, 필터 칩, 초기화 버튼
- `StatusBadge`: 상태별 색상과 라벨
- `ToneBadge`: 성공/주의/위험/정보/중립 톤
- `MobileActionCard`: 현장 대응용 카드와 큰 액션 버튼 패턴
- `FieldToolbar`: 리스트 상단의 검색/필터/액션 정렬 패턴

## 5. 개선 우선순위

### P0: 현장 모바일 대응

1. 전문가 대시보드 모바일 점검 및 버튼/모달 사용성 보강
2. 스타트업 예약 화면 모바일 기본 보기 재검토
3. 행사 상세 진행 현황의 모바일 현장 모드 설계
4. staff 체크인/사진 업로드 화면 모바일 퍼스트 설계

### P1: 백오피스 데이터 운영성

1. 참가자 DB 페이지네이션/정렬/고급 필터
2. 행사 목록 테이블 전환
3. 알림 로그 테이블 전환 및 실패 필터
4. 참가자 지정 목록 테이블화
5. 상담일지/만족도/사진/행사알림 결과 화면의 검색·필터·30개 페이지네이션 적용

### P2: 디자인 시스템 정리

1. 라운드 값 기준 확정
2. 상태 색상 토큰/컴포넌트화
3. 필터 탭/칩/배지 패턴 통합
4. 카드 안 카드처럼 보이는 중첩 패널 정리

## 6. 결론

현재 서비스는 기능 구현은 상당히 진행되었고, 일부 핵심 화면은 이미 반응형을 고려하고 있다. 다만 운영 규모가 커질수록 백오피스 화면은 테이블/검색/필터/페이지네이션 중심으로 강화해야 한다.

반응형 UI의 핵심은 모든 화면을 모바일에 맞추는 것이 아니다. 현장에서 실제로 쓰는 화면은 모바일 친화적으로, 대량 데이터 관리 화면은 데스크톱 생산성 중심으로 나누어 설계하는 것이 이 서비스에 가장 적합하다.
